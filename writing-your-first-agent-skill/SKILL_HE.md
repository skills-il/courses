# איך כותבים את הסקיל הראשון שלכם

רוב המפתחים שמשחררים סקיל ראשון מקבלים אחת משתי תוצאות: סקיל קטן וממוקד שמותקן מהר, או SKILL.md מבולגן באורך 2000 מילים שה־LLM אף פעם לא יטען כי התיאור עמום מדי. ההבדל הוא לא כמות העבודה. ההבדל הוא להבין מה הסקיל באמת מבחינת ה־LLM ברגע הניתוב, ומה המפרט מאפשר לכם לומר.

הקורס הזה למפתחים שיודעים לכתוב קוד אבל עוד לא כתבו סקיל סוכן, או שכתבו אחד ורוצים להבין למה Claude / Cursor / Windsurf טוענים חלק מהסקילים בעוד אחרים יושבים בקטלוג בלי שימוש. שישה פרקים, דוגמאות אמיתיות מהקטלוג skills-il. בסוף תוכלו לשחרר סקיל משלכם (או שתהיה לכם תוכנית ברורה לסקיל שאתם עומדים לכתוב).

## פרק 1: מה זה סקיל באמת?

סקיל סוכן של Claude הוא היחידה הקטנה ביותר של שינוי התנהגות LLM שאפשר לשחרר: קובץ Markdown אחד עם YAML frontmatter, אופציונלית מלווה בקבצי reference ובסקריפטים, שה־LLM קורא לפי דרישה כשהתיאור שלו תואם לבקשת המשתמש. זה כל המודל. כל השאר (קטלוג, sync pipeline, ציון אמון, מספרי התקנות) הוא תשתית סביב אותו קובץ בודד.

### השוואה ארבע־כיוונית: סקיל מול MCP מול system prompt מול CLI

![סקיל לעומת MCP לעומת system prompt לעומת CLI](skill-vs-mcp-vs-cli.png)

הכי קל להבין מה סקיל הוא דרך מה שהוא לא.

| כלי | מה זה | מתי תרצו לכתוב אחד |
|---|---|---|
| **סקיל סוכן** | קובץ Markdown שה־LLM קורא לפי דרישה כדי ללמוד איך לטפל בסוג משימה ספציפי | כשאתם רוצים שה־LLM ידע איך לעשות משהו בלי שתחזרו על ההוראות בכל שיחה |
| **שרת MCP** | שירות רשת שה־LLM קורא אליו כדי לקבל מידע חי או לבצע פעולות עם state | כשאתם צריכים שה־LLM יקרא או יכתוב state חיצוני (DB, API, מערכת קבצים) |
| **System prompt** | הוראות קבועות שמוזרקות בתחילת שיחה | כשאתם רוצים התנהגות שחלה על כל האינטראקציות של עוזר, לא רק על משימות ספציפיות |
| **כלי CLI** | תוכנית שהמשתמש מריץ ישירות | המשתמש יודע בדיוק איזו פקודה הוא רוצה להריץ. אין צורך בניתוב בשפה טבעית |

בלבול נפוץ אחד: אנשים כותבים סקיל כשהם בעצם צריכים MCP. אם ה"סקיל" שלכם מסתכם ב"ה־LLM צריך לקרוא ל־API הזה ולהחזיר את ה־JSON", זה שרת MCP, לא סקיל. סקילים הם לאיך לעשות משהו (כללי החלטה, מבנה, שיקול דעת). MCP הם לגישה ל־state חי.

בלבול נפוץ שני: אנשים כותבים סקיל כשהם בעצם צריכים system prompt. אם אתם רוצים ש־Claude תמיד תדבר בקול המותג שלכם בכל שיחה, זה system prompt או GPT מותאם, לא סקיל. סקילים נטענים לפי דרישה, כשכוונת המשתמש תואמת לתיאור הסקיל.

### מפרט Claude Skills של Anthropic, בפסקה אחת

כל סקיל הוא תיקייה שמכילה לפחות קובץ `SKILL.md` עם YAML frontmatter. ה־frontmatter דורש שני שדות לפי המפרט של Anthropic: `name` (slug ב־kebab-case) ו־`description` (משפט או שניים שה־LLM קורא כדי להחליט אם לטעון את הסקיל). השדה `license` (בדרך כלל MIT) הוא קונבנציונלי בקטלוג skills-il, אם כי במפרט של Anthropic הוא טכנית אופציונלי. הגוף הוא Markdown רגיל, נטען ל־context של ה־LLM ברגע שהסקיל מוזעק. אופציונלית התיקייה יכולה לכלול תת־תיקיית `references/` (קבצים סטטיים שה־LLM קורא בעת הצורך) ותת־תיקיית `scripts/` (קוד שניתן להרצה שה־shell tool של הסוכן יכול להריץ). זה כל המפרט. כל מה שבקטלוג skills-il הוא superset של זה.

### ההרחבה של skills-il: metadata.json ותוכן דו־לשוני

המפרט של Anthropic מינימלי. הקטלוג skills-il צריך יותר: שמות תצוגה דו־לשוניים, תגיות קהל ורמה, רשימות סוכנים נתמכים, פקודות התקנה. כדי לא להתנגש עם המפרט של Anthropic, skills-il שם את זה בקובץ נפרד בשם `metadata.json` ליד SKILL.md. תוכן ה־companion בעברית גר בקובץ מקביל בשם `SKILL_HE.md`. פרק 3 מכסה את שניהם בפירוט.

### מתי לא לכתוב סקיל (עץ ההחלטה)

לפני שאתם כותבים שורה אחת של SKILL.md, הריצו את הבדיקה הזו:

- **האם ה־LLM כבר עושה את זה טוב בלי סקיל?** אם תשאלו את Claude "פרמט לי את מספר הטלפון הישראלי הזה" והוא יעשה את זה נכון בניסיון הראשון, אתם לא צריכים סקיל לפרמוט טלפונים. סקילים הם למקרים שבהם התנהגות ברירת המחדל שגויה, חלקית או לא מתאימה תרבותית.
- **האם המשימה דורשת נתונים חיים?** אם התשובה משתנה בכל שעה (מחירי מניות, תגובות API ממשלתי, תנועה), סקיל הוא ההפשטה הלא נכונה. תכתבו שרת MCP.
- **האם המשימה היא חישובית טהורה עם תשובה אחת נכונה?** checksum פשוט או ממיר פורמט עשוי לעבוד יותר טוב כשורה אחת ב־scripts/ שנקראת מתוך סקיל גדול יותר, ולא כסקיל בפני עצמו.
- **אתם בטוחים שהקהל באמת צריך את זה?** סקילים עם פחות מ־5 התקנות אחרי 6 חודשים נוטים למות בקטלוג. דברו עם 3 משתמשים פוטנציאליים לפני שאתם כותבים.

הטעות הנפוצה בפרק 1: כתיבת סקיל במקום שבו MCP היה מתאים יותר. סימפטום: הגוף של הסקיל הוא בעיקר "קרא ל־URL הזה, פרסר את ה־JSON הזה, החזר את השדה הזה". פתרון: המירו לשרת MCP עם כלי אחד, רשמו אותו בספרייה, וה"סקיל" שלכם הופך לקריאת MCP בשורה אחת.

אם הנושא שלכם שורד את עץ ההחלטה, התקינו את הסקיל יוצר סקילים skills-il (skills-il-skill-creator) באמצעות `npx skills-il add developer-tools/skills-il-skill-creator` כדי לעשות scaffold למבנה התיקייה, והמשיכו לפרק 2 כדי ללמוד מה לשים ב־SKILL.md.

## פרק 2: האנטומיה של SKILL.md טוב

SKILL.md טוב עושה שלושה דברים במקביל: אומר ל־LLM מתי לטעון את עצמו (התיאור), מלמד את ה־LLM איך לבצע את המשימה (הגוף), ונותן ל־LLM מספיק דוגמאות כדי לטפל במקרי קצה בלי להעביר את הכדור למשתמש. אם אחד משלושת אלה חסר, הסקיל או שלא נטען (כשל תיאור), או עושה את הדבר הלא נכון (כשל גוף), או שואל את המשתמש יותר מדי שאלות הבהרה (כשל דוגמאות).

### שדות ה־frontmatter

![האנטומיה של SKILL.md עם הערות](skill-md-anatomy.png)

```yaml
---
name: israeli-id-validator
description: Validate Israeli national ID numbers (תעודת זהות) using the Luhn-style check-digit algorithm. Use when a user pastes a 9-digit ID and asks "is this valid", or when generating sample IDs for testing. Do NOT use for foreign passport numbers or business registration numbers.
license: MIT
---
```

- **`name`** חייב להיות kebab-case, חייב להתאים בדיוק לשם התיקייה, וזה ה־slug שה־LLM משתמש בו כדי לזהות את הסקיל באופן פנימי. בחרו משהו ספציפי: `israeli-id-validator`, לא `id-tools`. שדה חובה לפי המפרט.
- **`description`** הוא השדה הכי חשוב בסקיל שלכם. במארחים שמנתבים לפי תיאור (Claude Code זה המקרה הקנוני), ה־LLM קורא בעיקר את השדה הזה כדי להחליט אם לטעון את הסקיל ל־context. במארחים שבהם המשתמש בוחר סקיל ידנית מרשימה, התיאור הוא עדיין מה שהוא רואה ראשון. כך או כך, תתייחסו אליו גם כעותק שיווקי וגם כמפרט ניתוב. שדה חובה לפי המפרט.
- **`license`** בדרך כלל `MIT`. המפרט של Anthropic מתייחס אליו כקונבנציונלי ולא חובה במובן הקפדני, אבל הקטלוג של skills-il והכלים סביבו מצפים לו. כללו אותו.

### התיאור הוא קלט הניתוב

במארחים שמגלים סקילים אוטומטית לפי התיאור (Claude Code הוא המקרה הקנוני), ה־LLM סורק את התיאורים של הסקילים הזמינים כדי להחליט איזה (אם בכלל) לטעון. הוא לא קורא את הגוף, הוא לא קורא את ה־references/, הוא קורא רק את התיאור. במארחים שבהם המשתמש בוחר סקיל מרשימה (Claude Desktop, חלק מ־Cursor flows), התיאור הוא עדיין הסיכום הראשון שהוא רואה ובדרך כלל מה שמכריע אם הוא לוחץ. כך או כך, התיאור חייב:

1. לקרוא למשימה בשמה באופן ברור (בשתי השפות של המונח, אם יש שם בהקשר ישראלי)
2. להפעיל לפי דפוסי שפה טבעית שמשתמש באמת ישתמש בהם
3. לא לכלול מקרים שנראים דומים אבל דורשים סקיל אחר

הדפוסים "Use when..." ו־"Do NOT use for..." הופכים את זה לקונקרטי:

> "Validate Israeli national ID numbers (תעודת זהות) using the Luhn-style check-digit algorithm. Use when a user pastes a 9-digit ID and asks 'is this valid', or when generating sample IDs for testing. Do NOT use for foreign passport numbers or business registration numbers."

הסעיף "Do NOT" קריטי. בלעדיו, ה־LLM עלול לטעון את הסקיל הזה כשמשתמש שואל על דרכון רומני, ואז ינסה להפעיל את אלגוריתם ספרת הביקורת הישראלי וייכשל באופן מבלבל. עם סעיף "Do NOT", ה־LLM ינתב נכון למקום אחר.

### מבנה הגוף: היקף, כללים, דוגמאות, אנטי־דפוסים

הגוף הוא מה שה־LLM קורא אחרי שהוא החליט לטעון את הסקיל. הוא צריך 4 חלקים, פחות או יותר בסדר הזה:

1. **היקף** (פסקה אחת): מה בדיוק הסקיל הזה מכסה, ניסוח מחדש בדיוק רב יותר ממה שהתיאור אפשר.
2. **כללי החלטה** (העיקר): הלוגיקה אם־אז שה־LLM צריך להפעיל. השתמשו ברשימות תבליט, עצי החלטה, או דוגמאות עבודה. הימנעו מפרוזה.
3. **דוגמאות עבודה** (חובה): הראו את הקלט, את ההיגיון ואת הפלט. כסו את המקרה הטיפוסי ולפחות מקרה קצה אחד. בלי דוגמאות עבודה, ה־LLM ימציא את התנהגות הקצה.
4. **אנטי־דפוסים** (מומלץ): תגידו ל־LLM מה לא לעשות, גם אם זה נשמע מפתה. אנטי־דפוסים מונעים מצבי כשל שאחרת ייראו נכונים.

### דוגמה ללמידה: `israeli-id-validator`

התקינו את הסקיל מאמת תעודת זהות (israeli-id-validator) באמצעות `npx skills-il add developer-tools/israeli-id-validator` וקראו את SKILL.md שלו. זה סקיל קטן וממוקד (כ־150 שורות) עם היקף ברור, אלגוריתם דטרמיניסטי ב־scripts/, ודוגמאות עבודה קונקרטיות. זו הדוגמה הכי נקייה ל"איך סקיל ראשון קטן צריך להיראות".

הטעות הנפוצה בפרק 2: כתיבת תיאור עמום מדי. "סקיל לדברים ישראליים" נטען בכל שיחה בהקשר ישראלי ולא עוזר לאף אחת מהן. בחרו משימה אחת. תקראו לה בשמה בדיוק. הוסיפו את הדפוסים "Use when..." ו־"Do NOT use for...".

## פרק 3: metadata.json + תוכן דו־לשוני (ההרחבה של skills-il)

המפרט של Claude Skills של Anthropic מכסה את `SKILL.md` ואת תת־התיקיות האופציונליות `references/` + `scripts/`. הקטלוג skills-il צריך יותר: שמות תצוגה דו־לשוניים, תגיות קהל, רשימות סוכנים נתמכים, פקודות התקנה, מעקב גרסה. במקום להרחיב את ה־YAML frontmatter (מה שהיה מתנגש עם המפרט במעלה הזרם ושובר את הפרסר ה־YAML המחמיר של Claude Desktop), skills-il שם את המטא־דאטה הזה בקובץ נפרד: `metadata.json`. תוכן ה־companion בעברית גר בקובץ מקביל: `SKILL_HE.md`.

### למה metadata.json במקום YAML מקונן

הפרסר ה־YAML הקפדני של Claude Desktop דחה את המפתח המקונן `metadata:` ש־skills-il השתמש בו במקור בתוך ה־SKILL.md frontmatter, וסירב לטעון את הסקיל בכלל. הפתרון היה למשוך את כל מה שספציפי ל־skills-il לקובץ `metadata.json` נפרד ליד SKILL.md. עכשיו Claude Desktop (שקורא רק את ה־SKILL.md frontmatter) נטען חלק, וה־pipeline של הקטלוג (שקורא את metadata.json) מקבל את כל המידע הדו־לשוני ונתוני הקטלוג שהוא צריך. ה־frontmatter נשאר מינימלי: `name`, `description`, `license`, וזהו.

### שדות metadata.json

![מבנה metadata.json עם השדות החשובים](metadata-json-structure.png)

```json
{
  "name": "israeli-id-validator",
  "version": "1.2.0",
  "license": "MIT",
  "display_name": {
    "he": "מאמת תעודת זהות",
    "en": "Israeli ID Validator"
  },
  "display_description": {
    "he": "אמתו תעודת זהות ישראלית בעזרת אלגוריתם ספרת הביקורת. שמיש לטפסים, נתוני בדיקה, וזיהוי תעודות לא תקינות לפני שליחה.",
    "en": "Validate Israeli national ID numbers using the check-digit algorithm. Useful for forms, test data, and catching bad IDs before submission."
  },
  "audience": "developers",
  "level": "beginner",
  "tags": {
    "he": ["תעודת זהות", "אימות", "ישראל"],
    "en": ["israeli-id", "validation", "checksum"]
  },
  "supported_agents": ["claude-code", "cursor", "windsurf", "claude-desktop"]
}
```

- **`display_name`** הוא הכותרת הדו־לשונית שמופיעה ב־UI של הקטלוג. עברית קודם.
- **`display_description`** הוא העותק השיווקי הדו־לשוני. הגרסה העברית חייבת להיקרא כעברית טבעית, לא כתרגום (פרק 6 מרחיב).
- **`audience`** הוא אחד מ: `developers`, `non-technical`, `professionals`, `mixed`. קובע את פילטר הקטלוג ואת ברירת המחדל של רגיסטר הכתיבה.
- **`level`** הוא אחד מ: `beginner`, `intermediate`, `advanced`.
- **`tags`** חייב לכלול מערכים מקבילים בעברית + אנגלית. הקטלוג מסנן לפי התגיות באנגלית. המערך בעברית הוא לתצוגה ב־UI העברי.
- **`supported_agents`** חייב להשתמש ב־slug הקנוני: `claude-code` (לא `claude`), `gemini-cli` (לא `gemini`), `cursor`, `windsurf`, `claude-desktop`. slug שגוי יוצג כעיגול אייקון אפור ריק על הכרטיס (אין ולידציה ב־sync; בדקו את הכרטיס שלכם ב־dev לפני שאתם דוחפים).

### SKILL_HE.md: ה־companion בעברית

כל סקיל ב־skills-il משחרר תרגום עברי בשם `SKILL_HE.md` ליד `SKILL.md`. שני הקבצים חולקים את אותה צורת YAML frontmatter (עם טקסט תיאור בעברית), אותו מבנה גוף (כותרות פרקים מתורגמות), ואותן הפניות תמונה.

טבעיות העברית היא הגורם הבודד הגדול ביותר שמבדיל בין סקילים עם מספר התקנות גבוה לבין הזנב הארוך. הטעות הכי נפוצה כאן היא לכתוב SKILL.md, להריץ אותו דרך מתרגם LLM, להדביק את הפלט ל־SKILL_HE.md ולשחרר. התוצאה נקראת כמו אנגלית בעברית: סדר המילים שגוי, מלא calques ("שכבת האסטרטגיה" עבור "the strategy layer", "שתי שאלות עוגן" עבור "two anchor questions"), אי־התאמות מין צצות ("זו לא ייעוץ" כשצריך "זה לא ייעוץ"). קוראים ישראלים מזהים את זה תוך שנייה.

הפתרון: אחרי הטיוטה הראשונה של ה־LLM, שדובר עברית ישראלי טבעי (או pass של rewriter ייעודי) יערוך אותה לטבעיות. הפנו לסגנון של ערכי כל זכות בשפה פשוטה, הסברים ב־TheMarker, או בלוגים ישראליים פופולריים בתחום שלכם.

### הפניה צולבת לסקילים אחרים בפרוזה עברית

כשהגוף של SKILL_HE.md שלכם צריך להפנות לסקיל אחר (למשל, להמליץ על סקיל קשור שהמשתמש אולי ירצה גם), השתמשו בשם התצוגה העברי ולאחריו ה־slug בסוגריים, ולא ב־slug kebab-case חשוף בתוך backticks. דוגמה:

- שגוי (עברית): `התקינו את הסקיל israeli-pension-advisor`
- נכון (עברית): `התקינו את הסקיל יועץ פנסיה ישראלי (israeli-pension-advisor)`

הצורה הנכונה שומרת על זרימת הקריאה בעברית ובמקביל שומרת על ה־slug הקריא למכונה לצורך בדיקת הצימוד של הקטלוג. תוכן באנגלית משתמש במוסכמה של slug חשוף בתוך backticks. רק עברית צריכה את הטיפול של שם תצוגה ועוד slug.

### דוגמה ללמידה: `hebrew-seo-geo-toolkit`

התקינו את הסקיל ערכת כלים לקידום אתרים ו-GEO בעברית (hebrew-seo-geo-toolkit) באמצעות `npx skills-il add marketing-growth/hebrew-seo-geo-toolkit` והשוו את ה־SKILL.md, SKILL_HE.md ו־metadata.json שלו. זה סקיל דו־לשוני עם מספר התקנות גבוה (220+) שעבר איטרציות לפי משוב משתמשים. התיאור בעברית נקרא טבעי, התגיות מקבילות כראוי, והגוף מכסה דאגות SEO ו־GEO ספציפיות לישראל שכלי SEO גנרי לא היה מטפל בהן. זו ההפניה ל"איך נראה סקיל דו־לשוני באיכות גבוהה".

הטעות הנפוצה בפרק 3: תרגום מילולי מאנגלית לעברית שנקרא שבור לקוראים ישראלים. הפתרון: כתבו את SKILL_HE.md כאילו הוא המקור, לא התרגום. חשיבה ראשונה בעברית מייצרת סקילים טובים בהרבה.

## פרק 4: references/ מול scripts/ (מתי משתמשים בכל אחד)

תיקיית סקיל יכולה לכלול שתי תת־תיקיות אופציונליות: `references/` ו־`scripts/`. הן משרתות מטרות שונות ובחירה לא נכונה בין השתיים היא סיבת דחייה תכופה. הכלל פשוט: `references/` לתוכן סטטי שה־LLM קורא, `scripts/` לקוד שניתן להרצה שה־LLM מריץ. החלק הקשה הוא לזהות באיזה צד של הקו התוכן שלכם נופל.

### references/: קבצים סטטיים שה־LLM קורא לפי דרישה

`references/` מחזיק קבצים שה־LLM יכול לקרוא כשהוא צריך אותם: הסברים ב־markdown, טבלאות חיפוש JSON, נתוני דוגמה, קבצי תבנית, מפרטי תמונה. ה־LLM קורא אותם דרך כלי file-read. הוא לא מריץ אותם.

שימושים טיפוסיים:

- **טבלאות חיפוש**: מערך של קידומות אזור ישראליות ממופות לערים, מאוחסן כ־JSON, נמשך כשה־LLM צריך לזהות את מקור מספר טלפון
- **תוכן הפניה ארוך**: הסבר 2000 מילים על תהליך הגשת מסים בישראל, נשמר מחוץ ל־SKILL.md (שצריך להישאר קצר ומנותב היטב) ונמשך פנימה רק כשה־LLM מגיע לשאלה ספציפית
- **תבניות**: חוזי דוגמה, טפסי דוגמה, boilerplate שה־LLM מתאים אישית למשתמש
- **מפרטי תמונה**: prompts ליצירת איורים, נשמרים בנפרד מהגוף

ערך טוב ב־`references/` הוא משהו שה־LLM צריך רק לפעמים, שטעינה שלו ל־context מראש הייתה מבזבזת tokens.

### scripts/: קוד שניתן להרצה שה־LLM יכול להריץ

`scripts/` מחזיק קוד שהסוכן יכול להריץ: מאמתים, פרסרים, חישובים דטרמיניסטיים, קריאות רשת. הסוכן מזעיק סקריפטים דרך ה־shell tool של המארח (Bash ב־Claude Code, המקבילה במארחים אחרים), קורא את stdout/stderr, ומשלב את הפלט המעובד בתגובה שלו. אין מנגנון מובנה של "function-call" לסקריפטים במפרט. סקריפטים הם executables פשוטים שהסוכן מריץ וקורא.

שימושים טיפוסיים:

- **חישוב דטרמיניסטי שה־LLM לא צריך לעשות שוב ושוב**: אלגוריתם ספרת ביקורת Luhn לתעודות זהות ישראליות (אחרת ה־LLM היה גוזר אותו מחדש באופן לא עקבי)
- **פרסרים**: פונקציה שמקבלת טקסט גולמי של תלוש משכורת ישראלי ומחזירה פירוט JSON מובנה
- **חיפושים חיצוניים שדורשים מפתח API**: פונקציה שקוראת ל־API ממשלתי כדי לחפש מידע על רישום עסק
- **קריאות רשת**: scrape של טבלה ציבורית, קריאה ל־RSS feed

סקריפטים חייבים להיות עצמאיים (להצהיר על כל התלויות בראש הסקריפט), לכלול הערת usage ברורה בראש, ולכתוב ל־stdout בפורמט צפוי שה־LLM יכול לפרסר.

### עץ ההחלטה

![עץ החלטה: references לעומת scripts](references-vs-scripts-tree.png)

כשיש לכם תוכן שתומך בסקיל, שאלו:

1. **האם זה חישוב דטרמיניסטי שה־LLM עלול לגזור מחדש לא נכון?** → `scripts/`
2. **האם זה דורש מפתח API, קריאת רשת או כתיבה למערכת קבצים?** → `scripts/`
3. **האם זה חיפוש סטטי, תבנית או הסבר ארוך שה־LLM צריך רק לפעמים?** → `references/`
4. **האם זה כלל קצר שתמיד רלוונטי או דוגמה?** → inline בגוף `SKILL.md`
5. **האם זה כלל ארוך שתמיד רלוונטי וה־LLM חייב לדעת תמיד?** → כנראה לא צריך להיות בסקיל. שקלו system prompt או סקיל אחר שמכסה אותו.

### גודל קובץ ומבנה

שמרו על קבצי `references/` מתחת ל־5000 מילים כל אחד (כ־20KB). אם אתם צריכים יותר, פצלו לקבצים מרובים ותנו ל־LLM לבחור את הנכון. קבצי `scripts/` צריכים להיות עצמאיים: סקריפט יחיד מתחת ל־200 שורות קל לתחזוקה. כל דבר גדול יותר כנראה צריך להיות מפוצל או להפוך לחבילה אמיתית.

הימנעו מ:

- הדבקת טבלת חיפוש בת 50 שורות לגוף SKILL.md. העבירו אותה ל־`references/<table-name>.json` והפנו אליה מהגוף ("see references/area-codes.json").
- כתיבת סקריפט Python בתוך גוף SKILL.md בתוך code fence. העבירו אותו ל־`scripts/<name>.py` והפנו אליו.
- שמירת קבצים בינאריים (PNG, ZIP) ב־`references/`, אלא אם הם נתוני דוגמה שה־LLM בודק ספציפית. תמונות הקטלוג גרות במונורפו של האתר, לא בתיקיית הסקיל.

### דוגמה ללמידה: scripts/ של `israeli-id-validator`

הסקיל מאמת תעודת זהות (israeli-id-validator), שמותקן באמצעות `npx skills-il add developer-tools/israeli-id-validator`, משחרר קובץ TypeScript קטן ב־`scripts/` שמיישם את אלגוריתם ספרת הביקורת Luhn לתעודות זהות ישראליות. הגוף של SKILL.md מתאר מתי להשתמש באלגוריתם. המתמטיקה הדטרמיניסטית עצמה גרה בסקריפט. זו דוגמה קלאסית ל"חישוב דטרמיניסטי שייך ל־scripts/".

לדוגמה מנוגדת, התקינו את הסקיל מפרמט טלפונים ישראלי (israeli-phone-formatter) באמצעות `npx skills-il add developer-tools/israeli-phone-formatter` ולמדו איך הוא משתמש ב־`references/` כדי לאחסן את טבלת החיפוש של קידומות ישראליות (סלולר, קווי, שירות מיוחד) בלי לשבץ אותן בגוף.

הטעות הנפוצה בפרק 4: הדבקת טבלת חיפוש ארוכה לגוף SKILL.md במקום ל־`references/`. סימפטום: הגוף הוא 80 אחוז טבלה ו־20 אחוז כללי החלטה ממשיים. פתרון: העבירו את הטבלה לקובץ JSON ב־`references/` והפנו אליה מהגוף. יעילות tokens משתפרת, והגוף חוזר להיות קריא כלוגיקת החלטה.

## פרק 5: שחרור לקטלוג skills-il

כתבתם SKILL.md, SKILL_HE.md, metadata.json, ו (אופציונלית) references/ ו־scripts/. הסקיל עובד מקומית כשאתם מתקינים אותו עם `npx skills-il add <category>/<slug>`. עכשיו אתם צריכים בפועל לשחרר אותו לקטלוג. הפרק הזה עובר על תהליך השחרור מקצה לקצה.

### שלב 1: בחרו את ה־repo הקטגורי הנכון

![תהליך הפרסום: מסקפולד עד ציון אמון](publication-flow.png)

skills-il מארגן סקילים ל־12 repos קטגוריים תחת הארגון `skills-il` ב־GitHub. כל repo הוא תיקייה נפרדת ניתנת לפרסום ב־npm:

| קטגוריה | מתי להשתמש |
|---|---|
| `tax-and-finance` | מסים ישראליים, פנסיה, ביטוח לאומי, בנקאות |
| `developer-tools` | מאמתים, מפרמטים, כלי scaffold, כלי dev |
| `government-services` | טפסים, אינטראקציות עם סוכנויות, תהליכים רשמיים |
| `legal-tech` | חוזים, ציות, רגולציות |
| `security-compliance` | אודיטים ביטחוניים, GDPR / תיקון 13 |
| `communication` | אימייל, social, מסרים |
| `marketing-growth` | SEO, פרסומות, אסטרטגיית תוכן |
| `accounting` | הנהלת חשבונות, שכר, דוחות פיננסיים |
| `food-and-dining` | מסעדות, כשרות, משלוחי אוכל |
| `health-services` | קופות חולים, סל בריאות, ניווט בריאות |
| `education` | משאבי למידה, הכנה למבחנים |
| `localization` | NLP עברי, תרגום, תאריכים, יחידות |

בחרו את הקטגוריה היחידה שמתאימה הכי טוב. אם סקיל פורש בין שתי קטגוריות, בחרו את זו שהמשתמשים יחפשו בה ראשונה, לא את זו עם יותר סקילים.

### שלב 2: scaffold דרך skills-il-skill-creator

הריצו את הסקיל יוצר סקילים skills-il (skills-il-skill-creator) (אתם אמורים כבר להתקין אותו אחרי ההמלצה של פרק 1). הוא עובר איתכם על בחירת קטגוריה, מתן שם slug, frontmatter SKILL.md, יצירת metadata.json, stub של ה־companion בעברית, ומבנה התיקייה. הפלט הוא תיקייה מלאה ומוכנה ל־commit.

### שלב 3: ולידציה מקומית לפני push

לפני שאתם דוחפים, עברו על רשימת הבדיקות הבסיסיות לתיקיית הסקיל. ה־CI של ה־repo הקטגורי מריץ ולידטור. הרצת המקבילה מקומית קודם חוסכת round-trip. רשימת המינימום:

- ל־YAML frontmatter יש `name`, `description`, ו־`license`. אין מפתחות מקוננים.
- `metadata.json` הוא JSON תקין (`jq -e . metadata.json`) ותואם לסכמת הקטלוג של skills-il.
- ל־`SKILL.md` ול־`SKILL_HE.md` יש מבנה סעיפים מקביל (אותן כותרות פרקים מתורגמות).
- אין em dashes בשום מקום (עשו grep לשלושת הקבצים עבור תו ה־em-dash U+2014. הפלט חייב להיות ריק).
- שדה ה־`name` שווה לשם הבסיס של התיקייה (`jq -r '.name' metadata.json` תואם ל־`basename "$(pwd)"`).
- כל קובץ שמופנה מ־SKILL.md ל־`references/` או ל־`scripts/` באמת קיים בתיקייה.

אם יש לכם סקריפט ולידציה מקומי ב־repo הקטגורי (למשל `scripts/validate-skill.sh`), הריצו אותו. אחרת הבדיקות למעלה מספיקות כדי לתפוס את רוב דפוסי הדחייה.

### שלב 4: push ל־repo הקטגורי + פתיחת PR

תורמי קהילה: עשו fork ל־repo הקטגורי, push לתיקיית הסקיל ב־branch של פיצ'ר, ופתחו pull request. מתחזקי skills-il סוקרים וממזגים.

חברי הארגון עם גישה ישירה: push ל־master ב־repo הקטגורי. הגנת branch דורשת ביקורת PR כברירת מחדל. תפקיד admin יכול לעקוף לתוספות שגרתיות.

### שלב 5: ה־sync pipeline אוסף את הסקיל החדש

ה־sync pipeline (`pnpm sync-skills` ב־frontend של skills-il) רץ ב־cron יומי ו־on-demand דרך GitHub Actions. הוא עובר על כל repo קטגורי, קורא את SKILL.md + metadata.json, ועושה upsert לכל סקיל ל־DB של הקטלוג ב־Supabase. ב־sync הראשון, הסקיל שלכם מופיע בקטלוג עם `is_published=false`.

### שלב 6: admin משחרר את הסקיל דרך לוח המחוונים של ה־admin

`is_published` הופך ל־true דרך לוח המחוונים של ה־admin (או דרך `PATCH /v1/admin/skills/<slug>` עם ה־Supabase JWT הנכון). ה־admin סוקר את ערך הקטלוג, את ציון האמון, את התוכן שנוצר אוטומטית (שדות התוכן המועשרים מאוכלסים ב־pass העשרה נפרד), ומאשר או דוחה פרסום.

### שלב 7: pipeline ציון האמון מדרג את הסקיל שלכם

במקביל לפרסום, pipeline של ציון אמון סורק את ה־GitHub repo שלכם ואת תיקיית הסקיל לאיתותים על פני 3 שכבות:

- **קריטי (5 איתותים)**: spec compliant, secret scanning מופעל, code scanning מופעל, signed release, license מזוהה SPDX
- **מומלץ (8 איתותים)**: tag protection, branch protection, signed commits, SECURITY.md, MFA enforced, CODEOWNERS, dependabot clean, semver תואם tag
- **בונוס (2 איתותים)**: גיל release אחרון מתחת ל־180 ימים, tree SHA תואם HEAD

5 הקריטיים שעוברים מזכים בתג "Verified" ובשכבת אמון `verified`. כשל באחד מהקריטיים מוריד אתכם ל־`trusted`, `basic`, או `unverified`. שכבת הבונוס נותנת עד 10 נקודות מעל.

רוב התורמים בפעם הראשונה נוחתים ב־`basic` או `unverified` כי לא הפעילו secret scanning או לא חתמו על release. הפתרון פשוט: הפעילו secret scanning + push protection בהגדרות ה־repo, ואז תייגו את ה־release שלכם עם `git tag v1.0.0 && git push --tags` כדי להפעיל את workflow ה־signed release.

### שלב 8: איטרציה (version bumps, זרימת changelog)

כשאתם מעדכנים את הסקיל שלכם, עשו bump ל־`version` ב־metadata.json לפי semver (patch / minor / major). ה־sync pipeline אוסף את הגרסה החדשה בהרצה הבאה שלו. סעיף "what's new" של הקטלוג והתראות לעוקבים מונעים על ידי טבלת `skill_updates` במסד הנתונים, שמתמלאת על ידי flow של admin `update-skill`, ולא מקובץ `CHANGELOG.md` ספציפי לסקיל. סקילים לא צריכים לשחרר CHANGELOG.md (קורסים כן, סקילים לא). אם בא לכם changelog מקומי למעקב אישי, `CHANGELOG.md` זה בסדר אבל לא נדרש.

שכחה לעשות bump לגרסה כששוחררים שינויים היא טעות פרסום נפוצה. ה־sync pipeline לא דוחה את ה־push (הוא פשוט עושה upsert לכל מה שב־file), אבל מחוון ה"version" של הקטלוג נשאר ישן ול־flow של admin `update-skill` אין שום דבר חדש לפזר לעוקבים.

הטעות הנפוצה בפרק 5: push בלי לעדכן את version של metadata.json. תמיד עשו bump לגרסה כשאתם עורכים קובץ כלשהו בתיקיית הסקיל. אם ערכתם רק SKILL_HE.md (ליטוש תרגום), bump של patch (0.0.X). אם הוספתם כלי או פרק חדש, bump של minor. אם שיניתם את ההיקף של הסקיל או הסרתם פיצ'ר, bump של major.

לכלי שחרור שעושה scaffold נכון בפעם הראשונה, התקינו את הסקיל יוצר סקילים skills-il (skills-il-skill-creator) באמצעות `npx skills-il add developer-tools/skills-il-skill-creator`. הקורס נותן לכם את העקרונות. הסקיל מבצע את ה־workflow בלי שתצטרכו לזכור כל שדה.

## פרק 6: רשימת תיוג של 10 דקות לפני שחרור

הפרק הזה קצר בכוונה. זו רשימת התיוג שאתם מריצים לפני push. 10 הדקות שאתם משקיעים כאן חוסכות סבבים מרובים של הלוך ושוב מול pipeline הביקורת.

### 10 הסיבות הנפוצות לדחייה

1. **תיאור עמום מדי.** "סקיל לדברים ישראליים" מנותב לא טוב. השתמשו בדפוס "Use when X, Y, Z. Do NOT use for A, B".
2. **אין סעיף "Do NOT use for".** בלעדיו, ה־LLM טוען את הסקיל שלכם בהקשר הלא נכון ומייצר תשובה שגויה בביטחון מלא.
3. **calques בעברית ב־display_description או ב־SKILL_HE.md.** "שכבת האסטרטגיה", "שתי שאלות עוגן", סדר מילים מילולי מאנגלית. קורא ישראלי טבעי מאתר את זה תוך 5 שניות.
4. **ספים, מספרי טפסים או ציטוטי חוק מומצאים.** מדרגות מס שגויות ב־2026. מספרי טפסי ביטוח לאומי שגויים. הסוקר עושה cross-check מול מקורות ראשיים. המצאות נדחות בלי רחמים.
5. **slug לא תואם לשם התיקייה.** ה־sync pipeline אוסף את שם התיקייה. metadata.name חייב להיות זהה.
6. **אין דוגמת עבודה בגוף.** בלי דוגמאות, ה־LLM הוזה מקרי קצה.
7. **הפניות בקבצים ב־references/ או scripts/ שלא קיימים.** הוולידטור המקומי תופס את זה. תריצו אותו.
8. **slugs שגויים ב־supported_agents.** `claude` במקום `claude-code`, `gemini` במקום `gemini-cli`. מוצגים כעיגולי אייקון אפורים ריקים על הכרטיס.
9. **em dashes בכל מקום ב־SKILL.md, ב־SKILL_HE.md או ב־metadata.json.** מוסכמת הפרויקט היא אין em dashes, נקודה. השתמשו בפסיקים, סוגריים, נקודתיים.
10. **שכחה לעשות bump לגרסה ב־metadata.json אחרי עריכות.** ה־sync דורס תוכן בשקט אבל מחוון הגרסה של הקטלוג נשאר ישן ול־flow של admin `update-skill` אין שום דבר חדש לפזר לעוקבים.

### רשימת התיוג של 10 דקות לפני שחרור (מילולית)

![רשימת תיוג 10 דקות לפני שחרור](pre-publish-checklist.png)

הריצו אותן בסדר. כל אחת היא פקודת shell יחידה או קריאה של 30 שניות.

1. עשו grep לשלושת הקבצים אחרי תו ה־em-dash (U+2014). הפלט חייב להיות ריק.
2. `jq -r '.name' metadata.json` ואמתו שזה שווה ל־`basename "$(pwd)"`.
3. קראו את ה־`description` שלכם בקול. האם הוא עונה על "מתי ה־LLM צריך לטעון את זה?" במשפט אחד? האם יש לו סעיף "Do NOT use for"?
4. קראו את SKILL_HE.md בקול (או שדובר עברית טבעי יקרא אותו). האם זה נקרא כעברית, או כאנגלית מתורגמת?
5. בחרו מספר אחד, אחוז אחד או סכום בשקלים בגוף שלכם. פתחו את המקור הראשי שלו. אמתו שזה תואם.
6. בחרו את הפרק עם הכי הרבה לוגיקת החלטה. מצאו את דוגמת העבודה. גזרו אותה מחדש על נייר כדי לאמת את המתמטיקה.
7. רשמו את הקבצים ב־`references/` וב־`scripts/`. עבור כל אחד, עשו grep ל־SKILL.md כדי לוודא שיש אליו הפניה.
8. פתחו את הקטלוג המקומי (`pnpm dev` ב־frontend של skills-il, נווטו לקטגוריה שלכם) ואמתו שאייקוני הסוכנים הנתמכים מוצגים כלוגואים אמיתיים של פלטפורמה, לא עיגולים ריקים.
9. עשו bump ל־`version` ב־metadata.json (CHANGELOG.md אופציונלי לסקילים, בניגוד לקורסים).
10. הריצו את הוולידטור המקומי פעם אחרונה.

אם כל ה־10 עוברים, push. אם משהו נכשל, תקנו מקומית קודם.

### מבחן "האם תשובת ברירת המחדל של ה־chatbot הייתה טובה יותר?"

השאלה הכי חשובה לשאול לפני פרסום: האם המשתמש היה ירוויח יותר בלי הסקיל שלכם, כשהוא מקבל את תשובת ברירת המחדל של ה־LLM?

כמה סקילים נכשלים במבחן הזה בשקט. הסקיל מותקן. הוא מנותב נכון. ה־LLM נטען בצייתנות את הגוף. והתגובה גרועה יותר ממה ש־Claude היה אומר בלי הסקיל בכלל, כי מחבר הסקיל קודד מידע מיושן או פישט את הכלל יתר על המידה.

זה קורה כשהגוף של הסקיל ישן יותר מהמקור הבסיסי. סקיל שמצטט מדרגות מס משנה קודמת, את השכר הממוצע של השנה שעברה, או טבלת עמלות מלפני שינוי רגולטורי שקרה ביני לביני, גרוע יותר מאין סקיל בכלל. תשובת ברירת המחדל של ה־LLM, בלי הסקיל, לפחות הייתה שואבת מהידע הכללי שלו ומכירה בחוסר ודאות. הסקיל הישן מספק בביטחון מספר שגוי. סקילים חייבים להישאר מעודכנים. ה־flow של admin `update-skill` קיים בדיוק לסיבה הזו.

אם אתם לא יכולים להתחייב לשמור על הסקיל שלכם מעודכן כשהדומיין הבסיסי משתנה, אל תפרסמו אותו. בחרו נושא יציב (אלגוריתמים, כללים מבניים, דפוסים נצחיים) במקום נושא רגיש לזמן (שיעורים, מחירים, הצעות מוצר נוכחיות).

### מתי לשחרר סקיל מול שרת MCP

מסגרת אחרונה: אם ה"סקיל" שלכם הוא באמת "תגיד ל־LLM איך לקרוא ל־API הזה ולפרסר את התגובה הזו", אל תכתבו סקיל. תכתבו שרת MCP. ספריית ה־MCP של skills-il מקבלת הגשות קהילה באותו אופן כמו קטלוג הסקילים. שרתי MCP מטפלים ב־state חי טוב יותר, קל יותר לשמור אותם מעודכנים, והם משתלבים באופן טבעי עם סקילים שמטפלים בקבלת החלטות.

חלוקה נקייה:

- **סקיל**: איך לפרש את התגובה משרת MCP. מה לעשות עם הנתונים. כללי ההחלטה.
- **שרת MCP**: איך לאסוף את הנתונים. חוזה ה־API. ה־state הנוכחי.

כששניהם נדרשים (וזה רוב מקרי השימוש הישראליים), שחררו את ה־MCP כחבילה נפרדת ושהסקיל ימליץ עליו דרך פקודת ההתקנה בגוף.
